# 海克斯大乱斗数据源策略

## 1. 先纠正产品定位

当前项目不是斗魂竞技场 / Arena，而是：

**海克斯大乱斗助手**

对应页面和 SEO 应该围绕：

- Hex Brawl
- ARAM Hex Brawl
- 海克斯大乱斗
- 海克斯大乱斗英雄
- 海克斯大乱斗符文
- 海克斯大乱斗出装

不要再使用 `Arena Build`、`Arena Augments`、`Arena Decision Helper` 作为主定位，否则会和斗魂竞技场混淆。

## 2. 国外数据站现状

目前 Lolalytics、OP.GG、U.GG、LeagueOfGraphs 这类主流英文数据站更成熟的是：

- 排位 / 匹配
- ARAM
- Arena / 斗魂竞技场
- Champion build
- Item build
- Rune build

但它们基本没有公开完整、稳定、可抓取的“海克斯大乱斗”英雄统计页，至少不像排位和 Arena 那样有成熟结构。

这意味着：

- 英文 SEO 竞争可能更小。
- 数据供给更难。
- 如果能做出结构化页面，差异化会更明显。
- 首版不能依赖国外站点现成数据。

## 3. 掌上英雄联盟数据能不能对接

结论：

**业务上值得研究，技术上可能能抓到，但不建议 MVP 直接依赖掌盟内部接口。**

原因：

- 掌盟是腾讯官方 App，但没有发现面向第三方开发者公开的海克斯大乱斗数据 API。
- App 内部接口通常需要登录态、签名参数、设备参数、版本参数或风控校验。
- 接口结构可能随版本变化，稳定性无法保证。
- 未授权抓取可能涉及服务条款、版权、反爬和账号风控风险。
- 如果未来商业化或公开推广，数据来源不清会成为风险点。

所以不能把它当作“正式 API 对接”。更准确的说法是：

```text
掌盟数据可以作为研究参考，但不应作为 MVP 的生产依赖。
```

## 4. 不建议做的方案

### 4.1 直接抓 App 内部接口

不建议作为上线方案。

风险：

- 需要逆向请求签名。
- 可能触发风控。
- 接口随版本变化。
- 数据版权和使用授权不明确。
- 一旦接口失效，整站数据断供。

### 4.2 批量搬运掌盟截图数据

不建议公开上线时这样做。

风险：

- 数据更新成本高。
- 截图内容不可结构化复用。
- 可能涉及内容搬运。
- 无法解释数据来源和更新机制。

### 4.3 上线时展示来源不明的精确胜率

不建议。

原因：

- 用户会追问样本量、服务器、版本、时间窗口。
- 如果数据不是你自己计算或授权获取，很难保证可信度。
- 精确百分比会让产品看起来像数据站，但首版能力还不够。

## 5. 推荐 MVP 数据方案

### 5.1 首版只做“半结构化人工数据”

先人工维护 20 个英雄：

- 推荐符文
- 推荐出装
- 推荐技能加点
- 适合搭配
- 简短理由
- 更新时间
- 版本号

不展示精确胜率，不展示样本量。

页面表达为：

- 推荐
- 优先
- 强势
- 常见选择
- 适合场景

而不是：

- 第一胜率
- 最高胜率
- 官方统计
- 全服数据

### 5.2 使用掌盟作为“研究输入”

可以人工参考掌盟截图里的信息，例如：

- 哪些符文在当前版本排名靠前
- 哪些英雄经常出现在顶级英雄列表
- 某个英雄常见出装组合
- 某些符文的登场率排序

但前台不要写：

```text
数据来自掌上英雄联盟
```

除非拿到授权。

更稳的写法：

```text
Based on current patch trends and community-observed builds.
```

中文：

```text
基于当前版本常见打法和社区趋势整理。
```

### 5.3 半自动 OCR 工作流

如果你会持续从掌盟观察数据，可以做一个内部工具，而不是直接抓接口：

```text
掌盟截图
↓
OCR 识别符文名 / 英雄名 / 出装
↓
人工校对
↓
导入 JSON
↓
生成页面
```

优点：

- 不需要破解 App 接口。
- 数据录入速度比纯手工快。
- 仍然保留人工校对，避免错字和误识别。
- 上线页面不依赖实时外部接口。

缺点：

- 不是全自动。
- 需要每个 Patch 维护。
- 不能宣称完整统计覆盖。

## 6. 后期可行路线

### 路线 A：授权合作

最稳，但最慢。

目标：

- 联系数据方或内容方
- 获取允许展示的数据字段
- 明确更新频率和署名方式

适合产品有初步流量后再做。

### 路线 B：用户共创数据

让玩家提交：

- 英雄
- 符文
- 出装
- 使用体验
- 是否推荐

再人工审核。

优点：

- 合规压力小。
- 能形成社区内容。
- 比纯数据站更有差异化。

缺点：

- 早期冷启动慢。
- 需要审核机制。

### 路线 C：从公开对局 API 自己计算

如果 Riot / 腾讯有合规可用的数据入口，可以后期自己计算：

- 英雄胜率
- 符文登场率
- 符文胜率
- 出装胜率

但这不是 MVP，因为需要：

- 大量样本
- 稳定任务队列
- 数据清洗
- 版本隔离
- 防止样本偏差

## 7. 页面应该怎么改

首页继续保持：

- 标题：海克斯大乱斗助手
- 搜索：只搜英雄
- 语言切换：中文 / English / 日本語 / 한국어
- 下方：只显示英雄头像

英雄详情页建议做成：

```text
/champions/brand/hex-brawl
```

页面模块：

- 英雄头像 + 英雄名
- 海克斯大乱斗标签
- 符文推荐
- 技能加点
- 出装推荐
- 最佳拍档
- 一图流总结
- 更新时间
```

不要做复杂筛选，不要做排行榜，不要做登录。

## 9. MVP 结论

掌盟数据方向值得研究，但不适合作为第一版生产依赖。

第一版应该这样做：

1. 保留当前首页英雄搜索。
2. 改名为海克斯大乱斗助手。
3. 英雄详情页做一图流。
4. 首批 20 个英雄人工整理。
5. 掌盟截图作为内部参考输入。
6. 后台用 JSON 管理数据。
7. 后续再评估授权、OCR 或用户共创。

这样上线最快，风险最低，也不会因为接口不稳定导致项目停摆。
