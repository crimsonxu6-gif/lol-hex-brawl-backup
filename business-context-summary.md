# 海克斯大乱斗助手商业背景总结

更新时间：2026-06-17

## 1. 核心商业目标

当前项目的战略方向是：围绕《英雄联盟》“海克斯大乱斗”模式，打造一个轻量、移动端友好、可 SEO 获取流量的英雄攻略与决策辅助工具。

首阶段不追求复杂推荐引擎或全自动数据系统，而是用最短时间验证用户是否需要“海克斯大乱斗一图流助手”。

### 阶段目标

- 快速上线 MVP，验证搜索需求和页面停留质量。
- 优先完成首页英雄搜索、英雄详情页、出装推荐、海克斯推荐。
- 用 20 个高热度英雄作为首批内容池，避免一次性铺开低质量页面。
- 通过 Google SEO 获取第一批自然流量。
- 保持产品轻量，不引入登录、账号体系、复杂筛选、实时游戏内助手等高成本功能。

### 关键 KPI

首月验证指标：

- 20 个英雄详情页被 Google 收录。
- 100+ 自然搜索点击。
- 10+ 用户反馈或内容纠错。
- 10+ 收藏、分享或复访行为。
- 平均页面停留时间 > 45 秒。

中期增长指标：

- 扩展到 60 个英雄详情页。
- 形成稳定 Patch 更新流程。
- 通过 Search Console 识别高潜力关键词。
- 月活跃用户和自然搜索点击持续增长。

商业化暂不作为首阶段目标。后续在达到稳定自然流量后，再考虑 Google AdSense、内容合作或社区赞助。

## 2. 业务与产品梳理

### 产品定位

产品名称方向：

- 中文：海克斯大乱斗助手
- 英文：Hex Brawl Helper

产品不是“斗魂竞技场 / Arena 助手”，也不是通用 LOL 数据站，而是专注“海克斯大乱斗”这一细分模式。

### 核心产品形态

MVP 采用“首页英雄搜索 + 英雄详情页”的结构。

首页功能：

- 顶部标题：海克斯大乱斗助手
- 右上角语言切换：中文、英语、日语、韩语
- 中部英雄搜索栏，只搜索英雄
- 下方英雄头像网格
- 去除榜单、标签、复杂功能入口和额外小图标

英雄详情页功能：

- 英雄头像和英雄名
- 常规出装推荐
- 海克斯推荐
- 白银阶、黄金阶、棱彩阶分组
- 同阶级内按选取率从高到低排列
- 使用游戏内海克斯图标
- 去除技能加点、最佳拍档、S/A 评分等干扰信息

### 目标用户

核心用户：

- 海克斯大乱斗玩家
- 喜欢“一图流”攻略的轻度玩家
- 游戏中需要快速查看英雄推荐的移动端用户
- 对海克斯、出装、英雄适配不了解的回流玩家和休闲玩家

潜在用户市场：

- 中文用户：熟悉掌盟、微信、B 站、抖音等内容生态，偏好一图流和快速结论。
- 国际用户：依赖 Google 搜索、Reddit、Discord 等渠道，目前公开数据和攻略供给不足。

### 核心竞争力

- 切入细分：聚焦“海克斯大乱斗”，避开排位、ARAM、Arena 等成熟红海。
- 使用成本低：不要求玩家输入当前三个海克斯，不要求登录。
- 移动端优先：适合游戏中快速打开查看。
- 内容简洁：先看出装，再看海克斯；减少无关信息。
- SEO 机会：国外主流数据站对海克斯大乱斗完整页面覆盖不足，存在关键词空位。
- 可迭代：首版人工整理，后续可逐步接入 OCR、用户反馈、授权数据源。

## 3. 关键数据与结论

### 关于产品方向

最初 PRD 的核心方向是“决策助手”，但“用户输入 3 个海克斯，然后系统推荐一个”的方案在 MVP 阶段不适合直接做。

原因：

- 游戏中输入成本高。
- 海克斯名称存在中文、英文、简称、图标等多种识别问题。
- 实时推荐需要考虑装备、队友、阶段、已选海克斯等上下文，首版复杂度过高。
- 单点推荐容易被理解为“替玩家做决定”，存在表达和合规风险。

最终调整为：

```text
一图流决策辅助，而不是实时输入式推荐系统。
```

### 关于数据来源

已确认的结论：

- Lolalytics、OP.GG、U.GG、LeagueOfGraphs 等国外主流数据站，没有像排位、ARAM、Arena 那样公开完整的海克斯大乱斗英雄统计页面。
- 掌上英雄联盟中存在海克斯大乱斗相关数据，包括符文排行、英雄数据、出装、登场率、胜率等。
- 掌盟数据业务价值高，但未发现面向第三方开发者的公开海克斯大乱斗 API。
- 掌盟 App 内部接口不建议作为 MVP 生产依赖，因为可能存在登录态、签名参数、风控、接口变动和授权风险。

推荐数据策略：

- 首版人工整理 20 个英雄。
- 掌盟截图可作为内部研究参考，但不直接宣称数据来自掌盟。
- 后续可考虑 OCR 半自动录入：截图 -> OCR -> 人工校对 -> JSON -> 页面。
- 长期可探索授权合作、用户共创数据或合规数据源。

### 关于图标资源

英雄头像和装备图标：

- 使用 Riot Data Dragon。

海克斯图标：

- 可使用 CommunityDragon 的游戏数据资源。
- 已验证部分海克斯图标 URL 可访问，例如 `Infernal Conduit`、`Eureka`、`Goredrink`、`Ocean Soul`、`Ice Cold` 等。

### 关于页面实现

当前已完成的本地原型：

- 首页：`D:\LOL\index.html`
- 布兰德详情页：`D:\LOL\champions\brand\hex-brawl\index.html`
- 数据策略文档：`D:\LOL\hex-brawl-data-strategy.md`

已完成页面特性：

- 木质低饱和深色主题。
- 首页支持英雄搜索。
- 首页支持中文、英语、日语、韩语切换。
- 布兰德详情页支持中文、英语、日语、韩语切换。
- 布兰德详情页支持 `?lang=en` 等参数初始化语言。
- 布兰德详情页只保留出装和海克斯。
- 海克斯按阶级分组，并按选取率排序。
- 已去除 S/A 评分、技能加点、最佳拍档。

## 4. 待办与后续方向

### 短期待办

1. 继续完善布兰德详情页视觉。
   - 检查移动端真实手机显示。
   - 优化海克斯图标大小、间距和列表密度。
   - 优化出装模块信息层级。

2. 建立数据 JSON 结构。
   - 将布兰德页面中的出装和海克斯数据抽离为 JSON。
   - 设计统一字段：英雄、版本、语言、出装、海克斯阶级、选取率、图标路径。

3. 扩展首批英雄。
   - 建议首批 20 个英雄：
     Yasuo、Yone、Sett、Garen、Jinx、Ashe、Kai'Sa、Vayne、Zed、Katarina、Master Yi、Lee Sin、Lux、Ahri、Ezreal、Samira、Darius、Mordekaiser、Veigar、Kayle。
   - 同时补充布兰德，因为当前已作为详情页样板。

4. 统一多语言策略。
   - 中文优先。
   - 英文用于 SEO 和国际用户。
   - 日语、韩语先做 UI 文案和英雄名，海克斯名称可逐步补全。

5. 梳理 SEO 页面结构。
   - 首页：`/`
   - 英雄详情：`/champions/[slug]/hex-brawl/`
   - 后续集合页：`/hex-brawl-tier-list`、`/hex-brawl-augments`、`/hex-brawl-builds`

### 中期方向

1. 从静态 HTML 迁移到 Next.js。
   - 当前原型适合快速验证视觉和交互。
   - 后续若进入可上线版本，应转为 Next.js App Router。
   - 使用 SSG / ISR 生成 SEO 页面。

2. 建立内容更新流程。
   - 每个 Patch 更新一次。
   - 用人工维护或 OCR 校对生成 JSON。
   - 页面显示更新时间和 Patch。

3. 加入反馈机制。
   - 用户可标记“过期”、“出装不对”、“海克斯排序不对”。
   - MVP 阶段无需登录。

4. 增加分享图能力。
   - 每个英雄详情页可生成一张一图流攻略图。
   - 适合微信、Discord、Reddit、X 等渠道传播。

### 长期方向

1. OCR 半自动录入。
   - 从掌盟截图识别符文名、出装、登场率。
   - 人工校对后导入站点。

2. 用户共创数据。
   - 玩家提交推荐出装和海克斯。
   - 后台审核后进入页面。

3. 授权或合作数据源。
   - 若产品有流量，可尝试与数据方或内容方合作。

4. 轻量推荐引擎。
   - 不做首版。
   - 后续可让用户点击当前看到的几个海克斯，系统返回排序和理由。
   - 推荐应表达为“优先级参考”，不是“必须选择”。

## 5. 当前原则

产品继续遵循以下原则：

- 快速上线验证市场，不做过度设计。
- 先把英雄详情页做清楚，再考虑复杂功能。
- 先解决“看什么”而不是“算什么”。
- 不依赖未授权、不稳定的外部接口。
- 不展示来源不明的精确全服统计结论。
- 页面信息越少越好，优先服务移动端快速决策。
- 海克斯排序用选取率或人工优先级表达，但避免过度承诺。
